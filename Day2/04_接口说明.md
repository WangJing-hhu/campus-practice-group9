# 校园问答助手 Day2（实际 Day3）接口说明

## 一、统一技术约定

本文件用于统一前端、后端和数据库开发规范。任何成员不得自行修改接口路径、字段名称、角色值或状态值。如确需修改，必须先由项目经理统一通知全组。

### 运行端口

| 服务 | 地址 |
|---|---|
| React 前端 | http://localhost:5173 |
| Spring Boot 后端 | http://localhost:8081 |
| MySQL | localhost:3306 |
| 数据库名称 | campus_qa |

前端统一使用 `/api` 作为请求前缀，并通过 Vite 将 `/api` 代理到：

```text
http://localhost:8081
```

### 用户角色

| 后端值 | 前端显示 |
|---|---|
| admin | 管理员 |
| user | 普通用户 |

### 用户状态

| 后端值 | 含义 |
|---|---|
| 1 | 启用 |
| 0 | 禁用 |

### 删除方式

用户删除采用 MyBatis-Plus 逻辑删除：

| deleted | 含义 |
|---|---|
| 0 | 正常 |
| 1 | 已删除 |

不得直接物理删除用户数据。

## 二、统一返回格式

所有接口统一返回以下结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1780000000000
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| code | integer | 业务状态码 |
| message | string | 提示信息 |
| data | object / array / null | 业务数据 |
| timestamp | long | 响应时间戳 |

后端返回的业务状态码应与 HTTP 状态码保持一致。

---

## 三、统一状态码

| 状态码 | 含义 | 使用场景 |
|---|---|---|
| 200 | 请求成功 | 查询、登录、修改成功 |
| 201 | 创建成功 | 注册成功 |
| 400 | 参数错误 | 字段缺失、格式错误、非法角色 |
| 401 | 未认证 | 未携带 Token、Token 无效或过期 |
| 403 | 无权限 | 普通用户访问管理员接口、账号被禁用 |
| 404 | 资源不存在 | 用户不存在 |
| 409 | 数据冲突 | 用户名或邮箱已经存在 |
| 500 | 服务器异常 | 未处理的系统异常 |

## 四、用户注册接口

### 请求地址

```http
POST /api/user/register
```

### 是否需要 Token

不需要。

### 请求参数

```json
{
  "username": "student01",
  "password": "123456",
  "email": "student01@example.com"
}
```

### 参数规则

- username 不能为空，长度为 2—50 位。
- username 不能重复。
- password 不能为空，长度为 6—72 位。
- email 必须符合邮箱格式。
- email 不能重复。
- 密码必须经过 BCrypt 加密后再存入数据库。
- 新注册用户的默认角色为 `user`。
- 新注册用户的默认状态为 `1`。

### 成功响应示例

```json
{
  "code": 201,
  "message": "注册成功",
  "data": {
    "id": 2,
    "username": "student01",
    "email": "student01@example.com",
    "role": "user",
    "status": 1
  },
  "timestamp": 1780000000000
}
```

响应中不得出现 password 字段。

## 五、用户登录接口

### 请求地址

```http
POST /api/user/login
```

### 是否需要 Token

不需要。

### 请求参数

```json
{
  "username": "student01",
  "password": "123456"
}
```

### 登录规则

- 根据用户名查询用户。
- 使用 BCrypt 的 matches 方法验证密码。
- 用户不存在或密码错误时返回 401。
- 用户状态为 0 时返回 403。
- 登录成功后生成 JWT Token。
- Token 中至少包含 userId、username、role 和过期时间。

### 成功响应示例

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "JWT_TOKEN",
    "userId": 2,
    "username": "student01",
    "role": "user"
  },
  "timestamp": 1780000000000
}
```

## 七、管理员新增用户

### 请求地址

```http
POST /api/user
```

### 请求头

```http
Authorization: Bearer <token>
Content-Type: application/json
```

### 权限

仅管理员 `admin` 可以调用。

### 请求参数

```json
{
  "username": "student02",
  "password": "123456",
  "email": "student02@example.com",
  "role": "user",
  "status": 1
}
```

### 字段约定

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| username | string | 是 | 2—50位，唯一 |
| password | string | 是 | 6—72位 |
| email | string | 是 | 邮箱格式，唯一 |
| role | string | 是 | admin 或 user |
| status | integer | 否 | 1为启用，0为禁用，默认1 |

### 成功响应

HTTP 状态码：201

```json
{
  "code": 201,
  "message": "用户创建成功",
  "data": {
    "id": 3,
    "username": "student02",
    "email": "student02@example.com",
    "role": "user",
    "status": 1,
    "createTime": "2026-07-16T16:00:00"
  },
  "timestamp": 1780000000000
}
```

### 业务规则

- 仅管理员可以调用。
- 用户名或邮箱重复时返回 409。
- 密码必须使用 BCrypt 加密后存储。
- role 只能为 admin 或 user。
- status 只能为 0 或 1。
- 响应中不得包含 password 字段。
- 公开注册接口仍为 `POST /api/user/register`，且只能创建普通用户。

## 八、用户分页查询

### 请求地址

```http
GET /api/user/list
```

### 请求头

```http
Authorization: Bearer <token>
```

### 权限

仅管理员 `admin` 可以访问。

### 查询参数

| 参数 | 是否必填 | 默认值 | 说明 |
|---|---|---|---|
| page | 否 | 1 | 当前页码 |
| size | 否 | 10 | 每页条数 |
| keyword | 否 | 空 | 用户名或邮箱关键词 |
| role | 否 | 空 | admin 或 user |
| status | 否 | 空 | 1 或 0 |

### 完整请求示例

```http
GET /api/user/list?page=1&size=10&keyword=&role=&status=
```

### 成功响应中的 data

```json
{
  "records": [
    {
      "id": 2,
      "username": "student01",
      "email": "student01@example.com",
      "role": "user",
      "status": 1,
      "createTime": "2026-07-16 14:00:00"
    }
  ],
  "total": 1,
  "current": 1,
  "size": 10,
  "pages": 1
}
```

用户列表不得返回 password 字段。

## 九、修改用户信息

### 请求地址

```http
PUT /api/user/{id}
```

### 请求头

```http
Authorization: Bearer <token>
```

### 权限

仅管理员 `admin` 可以访问。

### 请求参数

```json
{
  "email": "new@example.com",
  "role": "user"
}
```

### 修改规则

- 允许修改邮箱。
- 允许修改角色。
- 禁止通过该接口修改用户名。
- 禁止通过该接口修改密码。
- role 只能是 `admin` 或 `user`。
- email 必须符合邮箱格式。
- 用户不存在时返回 404。

## 十、启用或禁用用户

### 请求地址

```http
PUT /api/user/{id}/status
```

### 请求头

```http
Authorization: Bearer <token>
```

### 权限

仅管理员 `admin` 可以访问。

### 请求参数

禁用：

```json
{
  "status": 0
}
```

启用：

```json
{
  "status": 1
}
```

### 业务规则

- status 只能是 0 或 1。
- 管理员不能禁用当前登录的自己。
- 被禁用的用户不能继续登录。
- 已登录用户被禁用后，再次验证 Token 时应提示账号不可用。

---

## 十一、逻辑删除用户

### 请求地址

```http
DELETE /api/user/{id}
```

### 请求头

```http
Authorization: Bearer <token>
```

### 权限

仅管理员 `admin` 可以访问。

### 业务规则

- 使用 MyBatis-Plus 逻辑删除。
- 将 deleted 更新为 1。
- 不物理删除数据库记录。
- 已删除用户不在普通用户列表中显示。
- 管理员不能删除当前登录的自己。

---

## 十二、JWT 请求约定

除注册和登录外，其他需要认证的接口统一携带：

```http
Authorization: Bearer <token>
```

后端 JWT 拦截器需要验证：

1. 是否存在 Authorization 请求头。
2. 请求头是否以 Bearer 开头。
3. Token 签名是否合法。
4. Token 是否过期。
5. Token 对应的用户是否存在。
6. 用户是否处于启用状态。
7. 管理员接口中用户角色是否为 admin。

---

## 十三、前端共同约定

- 登录成功后保存 Token 和用户信息。
- Axios 请求拦截器自动添加 Bearer Token。
- 收到 401 后清除本地登录状态并跳转登录页。
- 收到 403 时显示“无权限访问”或后端返回的 message。
- 前端隐藏管理员菜单只用于改善体验。
- 真正的权限判断必须由后端完成。
- 所有请求需要显示加载状态。
- 操作成功使用统一成功提示。
- 操作失败必须显示后端 message，禁止静默忽略异常。
- 所有成员必须严格使用本文件中的接口路径和字段。
